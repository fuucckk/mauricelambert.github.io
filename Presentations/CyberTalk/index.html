Cybersecurity Risks and Myths versus Real-World attacks

      

      

        

          
    
            
  • Financial Costs: Cyberattacks cost billions every year.
  • Reputation Damage: A single breach can ruin a brand's trust.
  • Privacy Threats: Data leaks expose personal information, compromising privacy.
    • 
          
    
            
  • High Costs: Cybersecurity solutions are expensive and often ineffective.
  • Insufficient Protection: Many defenses fail against threats.
  • False Sense of Security: Even with major investments, businesses remain vulnerable.
    • 
          
To demonstrate these challenges, we will explore a simulated cyberattack that exposes these vulnerabilities and debunks common cybersecurity myths.

When Security Practices Fail: Cracked Authentication Can Drain a Company's Funds

      

    

      

        
        
The Perfect Target: Small Startup, Big Funding, Single Point of Failure

      

      

        

          
Sylphora Dynamics is a small IoT & logistics startup based in Marseille.
Earlier this year, the company successfully raised €5 million in funding to accelerate its growth.
With fewer than 10 employees and no dedicated finance team, all banking and financial operations are managed directly by the CEO, Henri Brosquet, through his professional email account.
A group of attackers identifies this as a potential weakness.
Their objective: gain access to the CEO's email account in order to initiate discreet transfers from the company's investment funds.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
OSINT: The Public Data That Fuels Cyberattacks

      

      

        

          
OSINT (Open Source Intelligence) refers to the collection of publicly available information used to profile a company or individual.
For attackers, public data is the first weapon, it helps them understand structures, targets, and weaknesses.
    
            
  • Social networks like LinkedIn are a goldmine of intelligence: job titles, tools, partners, and even email patterns.
  • Press releases, websites, and GitHub commits often reveal technical and organizational insights.
    • 
          
In a real attack, everything begins with what's already publicly available.
Attackers don't guess, they research.

        

        
      

      
Sylphora Dynamics website

    

      

        
        
From Public Data to Password Cracking: The First Step of the Attack

      

      

        

          
After collecting public information, we identified two professional email addresses and a login portal accessible online.
The first realistic attack scenario: a brute-force attempt on these accounts.
Rather than random guessing, attackers often use password dictionaries based on the most common passwords used in a specific country or language.
Attackers don't need sophistication, just our bad habits.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
A New Path: From Failed Credentials to a Technical Weakness

      

      

        

          
    
            
  1. After the brute-force phase, some valid employee credentials are found, but not the CEO's.
  2. Using the newly found credentials, attackers try to log in, but reach a two-factor authentication wall.
  3. Looking for another entry point, they experiment with how the login form reacts to unexpected input.
  4. One of these tests triggers a server error, displaying technical details.
  5. Searching this error message online points to a database query issue, a classic sign of a SQL injection vulnerability.
  6. They gained access without needing a password and were able to directly exfiltrate all data.
    7. 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Mass Credential Extraction: Full User Database Compromised

      

      

        

          
After gaining access through the SQL injection, the attackers automate the process.
Using a simple script, they exfiltrate all user accounts, every hashed password, and all TOTP 2FA secrets stored in the database.
This gives them everything they need to rebuild the company's entire authentication system on their own machine, completely bypassing any security controls.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Targeted Password Bruteforce: User-Specific Wordlists Built from Public Data

      

      

        

          
After extracting the password hashes, attackers create personalized wordlists using public info: names, birth years, city, and data from the website.
Each user's list contains 36 million potential passwords.
On an old Linux i3 machine, generating and testing the entire 36-million-entry wordlist took only 101 seconds per user, even with a simple, non-optimized Python script.
However, parallel processing (multi-core CPUs) can reduce cracking time proportionally to the number of cores, modern GPUs can accelerate it by 10x to 100x, and ASICs by up to 100,000x.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Leveraging Breach Data: Identifying Passwords Reused Across the Internet

      

      

        

          
With employee names and corporate email formats, attackers can check publicly known data-breach collections to see whether any team members have had passwords exposed in past incidents.
If an employee reused the same or similar password across services, attackers may recover valid credentials without needing to guess or crack anything.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Highly Convincing Phishing: Impersonation, Punycode Domains & Realistic Email Cloning

      

      

        

          
With the CEO's email address, and even multiple compromised employee accounts, attackers could send highly convincing phishing emails (though this is not the method used in our demonstration).
In our scenario, they register a Punycode domain mimicking the real banking provider and clone a legitimate bank email to make the message fully credible.
The goal is to lure the CEO to a fake login page to capture his username, password, and 2FA code, or to deliver a crafted email exploiting a vulnerability in his mail application, or a malicious attachment capable of compromising his workstation.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Final Stage of the Attack: Credential Theft Through Malicious File Execution

      

      

        

          
    
            
  • Phishing leads to ZIP file download:
    • The ZIP contains a fake PDF disguised as a .lnk shortcut.
  • User clicks the deceptive .lnk file: 
    • A hidden command runs in the background while the PDF opens normally.
  • Malware extracts KeePass credentials: 
    • Modifies the KeePass configuration file.
    • Deploy a malware to send the credentials back to the attacker's Command and Control (C2) server.

Practical Cybersecurity Tips: Defend Against Attacks on Personal and Business Accounts

      

    

      

        
        
Essential Personal Cybersecurity Best Practices

      

      

        

          
    
            
  1. Always lock your workstation with a strong password
  2. Use a password manager and generate random, unique passwords
  3. Avoid public networks and do not handle confidential data in public places
  4. Keep all systems and applications regularly updated
  5. Use multiple email accounts (personal, professional, sensitive services)
  6. Never input confidential data into AI tools
  7. Limit installed apps and browser extensions
  8. Stay cautious with unexpected emails
  9. Do not follow the actions requested in suspicious emails
    10. 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Practical Cybersecurity Tips for Everyday Protection

      

      

        

          
    
            
  • Automatically clear cookies and site data when closing your browser
to reduce tracking and protect your privacy
  • Use private browsing mode for everyday online activity
to limit data retention and exposure
  • Verify sensitive or unusual requests through a separate channel
(phone, chat, in-person) and never use the contact details from the suspicious email
    • 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Key Professional Cybersecurity Recommendations

      

      

        

          
    
            
  1. Deploy EDR / EPP / Antivirus solutions to strengthen endpoint protection
  2. Avoid using AI to generate code unless strong code-review expertise is available
  3. Enforce two-factor authentication (2FA), especially for employees with lower cybersecurity awareness
  4. Build a DevSecOps team as early as possible to integrate security into development workflows
  5. Enable full disk encryption on all company devices to protect data in case of loss or theft
    6. 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Practical Cybersecurity Tips for Organizations

      

      

        

          
    
            
  1. Use Single Sign-On (SSO) to reduce the number of passwords employees must manage
  2. Limit mandatory password changes, but enforce strong password policies
  3. Create an email rule that moves all messages from your internal domain to a dedicated folder, making spoofed look-alike domain emails easier to detect

Conclusion: Breaking Common Cybersecurity Myths

      

    

      

        
        
Debunking Cybersecurity Myths: The Hard Truths

      

      

        

          
    
            
  • 2FA is often bypassed, phishing, compromised user devices, or server attacks can render 2FA ineffective
  • EDR / AV / SIEM cannot stop most attacks, attackers rely on legitimate administrative actions that evade detection and blocking
  • Password vaults are only as secure as the device, if the user's device is compromised, all stored credentials are at risk
  • An email from the correct address doesn't prove identity, sender addresses can be spoofed or impersonated
