CTF Overview

      

    

      

        
        
CTF: Capture The Flag Explained

      

      

        

          
CTF stands for Capture The Flag.

It is a competition where participants solve cybersecurity-related challenges to capture "flags."



These are gamified challenges on different cybersecurity topics.



Flags are like passwords when you find one, you can validate the challenge and move on to any challenge you want.



Each time you validate a challenge, you earn points.



A common format for a flag is:

flag{this_is_a_sample_flag}

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Why Participate in a CTF

      

      

        

          
CTFs offer a training environment usually fun to practice cybersecurity including offensive techniques while staying completely legal.



Challenges are generally created by people with strong skills sometimes experts which allows for advanced challenges on specific topics.



CTFs can prove your skills.



CTFs help you build your network.



Above all CTFs are meant to be fun.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Educational Objectives of CTFs in Cybersecurity

      

      

        

          
The goal of CTFs is to help you discover and practice certain technical aspects of cybersecurity.



Learn new concepts.



Discover better techniques or alternatives.



Work on your methodologies (although not always).



Challenge yourself and push your limits.



Improve your analytical skills and attention to detail.



Increase your problem-solving abilities.

Common CTF Formats

      

    

      

        
        
Jeopardy Format

      

      

        

          
The Jeopardy format is one of the most common in CTFs. It consists of a series of independent challenges grouped by category. Each solved challenge gives you points.



Participants choose a challenge to solve from the available categories.



Often played in teams but also possible solo.



The more complex the challenge the more points it gives.



Most challenges are independent from each other so you can choose the order in which you solve them and the categories you want to focus on.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Attack-Defend Format

      

      

        

          
The Attack-Defend format combines both offensive (attack) and defensive (protect) aspects.



Each team has its own server (or infrastructure) to defend while attacking other teams’ servers to capture flags and earn points.



Offensive and defensive: teams must secure their own systems while looking for vulnerabilities in other participants’ systems.



Some vulnerabilities can be partially mitigated but should not be completely blocked. You can kick attackers out when detected.



There is a sense of urgency and time management. Team organization is required. The strategy is more realistic. There is also a build phase.

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
King of the Hill Format

      

      

        

          
In a King of the Hill the main objective is to take control of a central service or resource and maintain that control as long as possible.



Once a team manages to take control of a service (for example a vulnerable server) it must defend it against attempts from other teams to take it back.



Each team must attack to gain control while also setting up defenses to avoid being kicked out.



There is a notion of quick hacking to defend your position as fast as possible and real-time team synchronization.

Solving Your First CTF Challenge

      

    

      

        
        
Methodology

      

      

        

          
    
            
  1. Identify the category, read the title and read the description.
  2. There are often hints and in some challenges you can figure out the idea behind it (vulnerability type, technology, etc.) even before starting.
  3. When you don’t know something: find and read the documentation (you usually don’t need to read everything).
  4. Search for resources, blogs or writeups that talk about the topic.
  5. Identify the problem and how to exploit it.
  6. If you don’t know how to do it, search either for writeups or for documentation of a tool that can help with the exploitation (sometimes there are examples).
  7. It is rare to have to find a zeroday. Most of the time it’s custom code, or known CVEs with exploitation documentation (not always with a ready-to-use exploit tool), or an issue that was reported but not fixed (not considered a vulnerability by the maintainers).
    8. 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
      

      

        
CTF Categories, Methods and Tools

      

    

      

        
        
Web Challenge

      

      

        

          
    
            
  1. Web application vulnerabilities (SQL Injection, XSS...)
  2. Identify if it is Web Client or Web Server challenge.
  3. Visit the website using a browser and identify client-server interactions and dynamic elements.
  4. If it is Web Server, identify the technologies used, including architecture, web server, language, framework, modules, etc.
  5. Check if it is custom development or just a product with little or no customization.
  6. If you have the source code, you can use SAST/DAST tools to find the vulnerability (don’t rely too much on them in CTFs).
  7. If it is a known technology, check if there is a HackTricks page or blogs about exploitation.
  8. Find and exploit the vulnerability.
    9. 
          
    
            
  • Resources: HackTricks, OWASP (Top 10 and more), blogs/writeups
  • Required tools: Web browser (Firefox), CyberChef
  • Highly recommended tools: BurpSuite (can be replaced by Python, more control, less user-friendly and no Java), jwt.io
  • Optional tools: Wireshark, Python with Selenium, Postman, Nuclei, skipfish, OWASP ZAP, Nikto
    • 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Cryptography Challenge

      

      

        

          
    
            
  1. Challenges related to cryptography and decryption
  2. Identify the algorithm used
  3. Check the implementation if it is a known algorithm and custom code
  4. Check the values of the constants (prime numbers or not, etc...)
  5. Check the security recommendations (key size, etc...)
    6. 
          
    
            
  • Tools: Python, John the Ripper, Hashcat
    • 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Networking

      

      

        

          
    
            
  1. Network analysis, Man-in-the-Middle (MITM) attacks
  2. Often considered a subcategory of Forensics, for example a pcap file containing exfiltrated data and the flag inside.
  3. Identify source/destination IPs (most of the time challenges are layer 3 and above in the OSI model)
  4. Identify the most used SourceIP/DestinationIP/DestinationPort combinations
  5. Identify the protocols and technologies used
  6. Identify encrypted protocols that are not accessible
  7. Filter the exchanges that seem interesting
  8. If you don’t know the protocol, look for writeups of challenges using the same protocol, known security issues, common misuses, or the RFC (if it exists)
  9. Reverse the content of interesting packets (Wireshark does a lot of the work), extract the data, identify the format of the extracted content
    10. 
          
    
            
  • Required tools: Wireshark, tshark, nmap, scapy, iptables (netfilter)
  • Optional tools: tcpdump, aircrack-ng, NetworkMiner
    • 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Forensics

      

      

        

          
    
            
  1. Digital trace analysis and information recovery
  2. Identify the type of challenge: disk analysis, memory dump analysis, network analysis (pcap, pcapng), live system analysis (to avoid if possible), malware reverse
  3. Identify the technology, the issue, the alert, the moment the issue was detected
  4. Disk and live system: identify if it is a server or a workstation, check logs, MFT or equivalent (possible time pivot), look for persistence methods, download folders, installed applications, modules/extensions/plugins, browser history/downloads/cookies, emails, services, bash/python history
  5. Memory dump: list processes (especially process tree with user, full executable path, and command lines), kernel modules, network connections
    6. 
          
    
            
  • Required tools: volatility (2 and 3), autopsy, Eric Zimmerman tools, Didier Stevens Suite
  • Recommended tools: Powershell, Python, bash
  • Optional tools: log analysis tools (search, filters, visualization)
    • 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Reverse Engineering

      

      

        

          
    
            
  1. Program decoding, binary analysis
  2. Identify the file type (usually executable but could be other formats like a Java binary or a .pyc)
  3. Identify the file structure (platform, architecture, high entropy, section sizes and names, imports and exports (rarer), etc...)
  4. Identify the programming language and compilation details (stripped or not)
  5. Check if the reverse can be simplified (unstrip for Go and Rust)
  6. Look at the function graph and identify possible obfuscation
  7. Spot known patterns or instruction repetitions
  8. Choose the tool depending on the format and the type of reverse to apply: static or dynamic reverse
    9. 
          
    
            
  • Required tools: Ghidra or IDA or BinaryNinja, gdb (with modules like GEF or Pwndbg), WinDBG, strings, JADX, dnSpy
  • Optional tools: PeBear
    • 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Binary Exploitation (Pwn)

      

      

        

          
    
            
  1. System vulnerability exploitation
  2. Identify the technology
  3. Identify security and compilation options
  4. Identify dangerous functions used
  5. Identify functions that can help with exploitation
  6. Identify memory allocations/frees
  7. Find the vulnerability
  8. Calculate how to exploit it or find interesting offsets through testing if needed
  9. Write the exploit and exploit the vulnerability remotely (usually you will need to perform a privesc using the remote exploit to retrieve the flag)
    10. 
          
    
            
  • Required tools: gdb (with modules like GEF or Pwndbg), pwntools
    • 
          

        

        
      

      
References: Github MauriceLambert, MauriceLambert WebSite.

    

      

        
        
Open Source Intelligence (OSINT)

      

      

        

          
    
            
  1. Public information gathering on the Internet
  2. Clearly identify the target and carefully list all the data you have before starting the search (it is easy to waste time on out-of-scope elements)
    3.

  • Optional tools: Maltego, theHarvester, GHunt, Epieos, Sherlock, holehe, maigret, Overpass turbo

Steganography

      

      

        

          
    
            
  1. Hidden information discovery in files (images, audio)
  2. Identify the file type
  3. Check the file format specifications
  4. Look at CTF challenges using this format
  5. Guess 😂

Active Directory (AD)

      

      

        

          
    
            
  1. Exploring Active Directory vulnerabilities
  2. Enumerate without user
  3. If enumeration fails: find a user (OSINT, ...), exploiting a service vulnerability, vulnerable Microsoft protocol
  4. Enumerate with user
  5. Identify privileged accounts
  6. Identify potentially vulnerable accounts (misconfigurations allowing bruteforce or credential reuse)
  7. Identify commonly exploited services
  8. Identify and audit risky configurations
  9. Identify possible paths to domain admin
    10. 
          
    
            
  • Required tools: BloodHound (SharpHound / python-bloodhound / RustHound), Impacket, Certipy, Evil-WinRM
  • Recommended tools: CrackMapExec (netexec), BloodyAD, Certify, Responder, Rubeus, RDP Client
  • Optional tools: Mimikatz / Pypykatz, PowerView, Kerbrute, PingCastle
  • Lateral movement tools: proxychains / revsocks / ligolo / Chisel, SSH Tunneling, Socat
    •

Hardware / Radio

      

      

        

          
    
            
  1. Security of hardware devices and radio communications
  2. Properly identify the hardware: components, pins/pinout, etc.
  3. Search for datasheets / technical documentation of relevant components
  4. Once interesting pins are identified, connect them using the identified debug protocol
  5. Dump the firmware using the protocol (JTAG, SWD, UART, SPI, I2C, TPM, ...)
  6. Unpack and reverse the firmware
  7. Find the vulnerability
  8. Exploit the vulnerability
    9. 
          
    
            
  • Tools: binwalk, Bus Pirate (?), OpenOCD (?), URJTAG (?), Ghidra, avrdude

How CTFs Are Organized

      

    

      

        
        
How CTFs Are Organized

      

      

        

          
    
            
  1. Where to find CTFs: CTFtime.org, Discord servers, cybersecurity communities, and social media channels regularly announce upcoming events.
  2. Participating without a team: You can play solo, or join a temporary team with other solo players especially common in onsite CTFs and in CTF-focused Discord servers.
  3. Format, duration, and access: Most CTFs are online and last between 24 and 72 hours. Challenges are usually accessible via a dedicated platform during the event.
  4. Rules: Always read and follow the official rules. Common rules include: no flag sharing, no attacking the CTF infrastructure, no brute-forcing, no fuzzing on the CTF platform, respect other teams
    5.
